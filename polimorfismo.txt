##UPCASTING - SUBIR EN LA JERARQUIA#########

El upcasting consiste en convertir un puntero o referencia de una clase derivada en un puntero o referencia de la clase base.

Es implicito, no necesita cast manual, y es seguro, porque todo objeto de la clase derivada tambien es un objeto de la clase base.

Ejemplo:


Perro p("Rex");

Animal* a = &p;   // upcasting implicito: Perro* -> Animal*


En este caso, p es un Perro, pero tambien puede tratarse como un Animal.

Luego:

a->hacerSonido();

Si hacerSonido() esta declarado como virtual en la clase base Animal, entonces C++ llama al metodo correcto segun el tipo real del objeto:

Perro::hacerSonido()

Pero si hacerSonido() no es virtual, entonces C++ usa el tipo del puntero, es decir Animal*, y llama a:

Animal::hacerSonido()

Por eso, para que exista polimorfismo dinamico, se necesita:


virtual
+
puntero o referencia a la clase base


## COMO FUNCIONA VIRTUAL INTERNAMENTE - LA VTABLE

Cuando una clase tiene al menos un metodo virtual, normalmente el compilador crea una tabla de funciones virtuales, llamada vtable.

Ademas, cada objeto de esa clase suele tener un puntero oculto llamado vptr, que apunta a la vtable correspondiente a su tipo real.

Ejemplo conceptual:


Objeto Perro:                  vtable de Perro:

+-------------------+          +----------------------------------+
| vptr ------------ |--------> | hacerSonido -> Perro::hacerSonido |
+-------------------+          | imprimir    -> Perro::imprimir    |
| nombre = "Rex"    |          +----------------------------------+
| raza = "Labrador" |
+-------------------+


Cuando haces:

Animal* a = new Perro("Rex");
a->hacerSonido();

ocurre algo parecido a esto:

1. El programa ve que a es un puntero a Animal.
2. Como hacerSonido() es virtual, no decide la funcion solo por el tipo del puntero.
3. Lee el vptr del objeto real.
4. Entra a la vtable del objeto real.
5. Encuentra que debe llamar a Perro::hacerSonido().
6. Ejecuta el metodo correcto.

Todo esto ocurre en tiempo de ejecucion, por eso se llama binding dinamico.

Nota: la vtable y el vptr son una forma comun de implementar virtual, aunque el estandar de C++ no obliga a que internamente se haga exactamente asi.

## SLICING - EL ERROR SILENCIOSO##################

El slicing ocurre cuando copias un objeto derivado por valor dentro de una variable de tipo base.

En ese caso, se pierde la parte especifica de la clase derivada.

Ejemplo:

Perro p("Rex");

Animal a = p;   // slicing

Aqui p es un Perro, pero a es solamente un Animal.

La parte de Perro se corta y se pierde en la copia.

Diagrama conceptual:

Objeto original Perro p:

+-------------------+
| parte Animal      |
| nombre = "Rex"    |
+-------------------+
| parte Perro       |
| raza = "Labrador" |
+-------------------+

Objeto Animal a despues de copiar:
+-------------------+
| parte Animal      |
| nombre = "Rex"    |
+-------------------+
La parte Perro se perdio.

Entonces, si haces:

a.hacerSonido();

se llamara al metodo de Animal, no al de Perro.

Incluso si el metodo es virtual, ya no importa, porque el objeto a ya no es un Perro. Ahora es solamente un Animal.

Por eso, esta funcion produce slicing:

void hacerHablar(Animal a) {
    a.hacerSonido();
}

Aunque le pases un Perro:

Perro p("Rex");
hacerHablar(p);


la funcion recibe una copia de tipo Animal, asi que se pierde la parte de Perro.

Regla importante:
El polimorfismo dinamico debe hacerse con punteros o referencias,
no pasando objetos por valor.

## POLIMORFISMO POR REFERENCIA###################

No siempre necesitas usar new, delete o heap para tener polimorfismo dinamico.
Tambien puedes usar referencias.
Ejemplo:

void hacerHablar(Animal& a) {
    a.hacerSonido();
}

Luego:

Perro p("Rex");

hacerHablar(p);


Aqui p vive en el stack, no en el heap.

Pero como la funcion recibe una referencia a Animal, ocurre upcasting:

```
Perro -> Animal&
```

Y si hacerSonido() es virtual, C++ llama correctamente a:

```
Perro::hacerSonido()
```

Esta version evita slicing y tambien evita tener que usar new y delete.

## POLIMORFISMO CON PUNTEROS EN EL STACK#######################

Tambien puedes tener objetos en el stack y guardar sus direcciones en punteros de tipo base.
Ejemplo:
```
Perro p1("Rex");
Gato g1("Michi");
Vaca v1("Lola");

Animal* animales[3];

animales[0] = &p1;
animales[1] = &g1;
animales[2] = &v1;

for (int i = 0; i < 3; i++) {
    animales[i]->hacerSonido();
}
```
Aqui los objetos reales estan en el stack:
```
p1
g1
v1
```
Pero el arreglo guarda punteros de tipo:
```
Animal*
```
Entonces ocurre upcasting:
```
Perro* -> Animal*
Gato*  -> Animal*
Vaca*  -> Animal*
```
Y gracias al metodo virtual, se ejecuta el metodo correcto para cada objeto.

Importante: como no se uso new, no se debe usar delete.

Incorrecto:
```
delete animales[i];   // MAL si los objetos estan en el stack
```
Correcto:
```
// No se hace delete.
// Los objetos se destruyen automaticamente al salir del scope.
```

## POLIMORFISMO CON PUNTEROS EN EL HEAP############

Tambien puedes crear los objetos en el heap usando new.
Ejemplo:
```
Animal* animales[4];

animales[0] = new Perro("Rex");
animales[1] = new Gato("Michi");
animales[2] = new Vaca("Lola");

    animales[i]->hacerSonido();
```

Como aqui si se uso new, entonces despues hay que liberar memoria con delete:
```
for (int i = 0; i < 4; i++) {
    delete animales[i];
    animales[i] = nullptr;
}
```

En este caso:
```
El arreglo de punteros puede estar en el stack.
Los objetos reales estan en el heap.
```
## DESTRUCTOR VIRTUAL########################
Si una clase base se usa de forma polimorfica y sus objetos derivados pueden ser 
destruidos mediante un puntero a la clase base, entonces el destructor de la clase 
base debe ser virtual.
Ejemplo problematico:
```
class Animal {
public:
    virtual void hacerSonido() {
        cout << "Sonido generico" << endl;
    }

    ~Animal() {
        cout << "Destructor de Animal" << endl;
    }
};

class Perro : public Animal {
public:
    ~Perro() {
        cout << "Destructor de Perro" << endl;
    }
};
```
Luego:

```
Animal* a = new Perro();

delete a;
```
El problema es que a es de tipo Animal*.
Si el destructor de Animal no es virtual, al hacer delete a, el comportamiento 
puede ser incorrecto. Puede que no se llame bien al destructor de Perro.
La forma correcta es:

```
class Animal {
public:
    virtual void hacerSonido() {
        cout << "Sonido generico" << endl;
    }

    virtual ~Animal() {
        cout << "Destructor de Animal" << endl;
    }
};
```
Ahora, si haces:
```
Animal* a = new Perro();

delete a;
```
se llama correctamente en este orden:
```
~Perro()
~Animal()
```
Esto es importante porque si Perro reserva memoria, abre archivos o usa recursos 
internos, su destructor debe ejecutarse correctamente.
Regla practica:
```
Si una clase base tiene metodos virtuales, casi siempre debe tener 
destructor virtual.
```
## ERRORES COMUNES EN POLIMORFISMO
1. Olvidar virtual en el destructor de la clase base.
Ejemplo:
```
Animal* a = new Perro();

delete a;   // peligroso si ~Animal no es virtual
```
Esto puede causar destruccion incorrecta del objeto.

2. Intentar crear un objeto de una clase abstracta.
Una clase abstracta es una clase que tiene al menos un metodo virtual puro.
Ejemplo:
```
class Animal {
public:
    virtual void hacerSonido() = 0;
};
```
Esta clase no se puede instanciar directamente:
```
Animal a;                // error
Animal* a = new Animal;  // error
```
Porque Animal es incompleta como objeto concreto. Sirve como base para otras clases.

3. Usar override cuando el metodo no existe en la clase base.
Ejemplo incorrecto:

```
class Animal {
public:
    virtual void hacerSonido() {}
};

class Perro : public Animal {
public:
    void hacerSonidoAnimal() override {
        cout << "Guau" << endl;
    }
};
```
Esto da error, porque hacerSonidoAnimal() 
no esta sobrescribiendo ningun metodo de Animal.
La forma correcta seria:
```
class Perro : public Animal {
public:
    void hacerSonido() override {
        cout << "Guau" << endl;
    }
};
```
override es util porque permite que el compilador revise si realmente estas 
sobrescribiendo un metodo virtual de la clase base.

4. Pasar objetos por valor y causar slicing.
Incorrecto:
```
void hacerHablar(Animal a) {
    a.hacerSonido();
}
```
Correcto con referencia:
```
void hacerHablar(Animal& a) {
    a.hacerSonido();
}
```
Correcto con puntero:
```
void hacerHablar(Animal* a) {
    a->hacerSonido();
}
```
## RESUMEN GENERAL
Upcasting:
Perro* -> Animal*
Seguro y automatico.

Downcasting:
Animal* -> Perro*
Mas delicado. No siempre es seguro.

Polimorfismo dinamico:
Requiere metodos virtuales y uso de punteros o referencias.

Sin virtual:
C++ decide el metodo segun el tipo del puntero o variable.

Con virtual:
C++ decide el metodo segun el tipo real del objeto en ejecucion.

Slicing:
Ocurre cuando copias un objeto derivado en una variable de tipo base.
Se pierde la parte derivada.

Destructor virtual:
Necesario cuando vas a borrar objetos derivados usando punteros a la clase base.
